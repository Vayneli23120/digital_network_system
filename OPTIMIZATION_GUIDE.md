# 高并发请求频率优化方案

## 问题分析

**现状问题：**
- 限流配置：60请求/分钟（所有请求统一限制）
- 快速切换导航时容易触发限流
- 页面加载时多个组件同时发起请求
- 无前端缓存，重复请求相同数据

**用户量增多时的风险：**
- 多用户同时访问会更快触发限流
- 重复查询造成服务器压力
- 响应时间增加，用户体验下降

---

## 解决方案

### 1. 分层限流策略（后端优化）

**新限流配置：**
```python
# GET 请求：120/分钟（读取操作较宽松）
# POST/PUT/DELETE：40/分钟（写入操作中等）
# 认证端点：10/分钟（防止暴力破解）
# 批量查询端点：放宽限制
```

**优化点：**
- 区分读取和写入操作的不同限流
- 清理逻辑优化，减少CPU占用
- 添加重试时间计算

**启用方式：**
将 `app/main.py` 中的限流中间件导入改为：
```python
from .shared.middleware.rate_limiter_v2 import RateLimitMiddleware
```

---

### 2. 前端数据缓存系统

**缓存策略：**
| 资源类型 | TTL | 说明 |
|---------|-----|------|
| devices | 30秒 | 设备列表 |
| deviceDetail | 1分钟 | 设备详情 |
| faults | 30秒 | 故障列表 |
| maintenance | 30秒 | 维修列表 |
| backups | 1分钟 | 备份列表 |
| logs | 10秒 | 日志数据 |
| dashboard | 1分钟 | 仪表板数据 |
| templates | 5分钟 | 配置模板（变动较少）|
| credentials | 5分钟 | SSH凭证（变动较少）|

**缓存层级：**
1. **内存缓存**：快速读取
2. **localStorage**：持久化，页面刷新后仍可用

**缓存失效：**
- 添加/编辑/删除操作后自动清除对应缓存
- 支持强制刷新（绕过缓存）
- 定期自动清理过期缓存

---

### 3. 请求管理优化

**已实施的优化：**

1. **请求取消**
   - GET 请求自动取消之前的相同请求
   - 防止竞态条件

2. **防抖（Debounce）**
   - 延迟 300ms 执行
   - 快速点击只触发最后一次

3. **自动重试**
   - 网络错误自动重试（最多3次）
   - 4xx 错误不重试

4. **节流（Throttle）**
   - 限制刷新按钮点击频率
   - 1秒内最多1次请求

---

### 4. 综合优化效果

**单用户场景：**
- 页面切换：从缓存读取，几乎瞬时
- 刷新操作：防抖后减少无效请求
- 重复查询：缓存命中，0延迟

**多用户场景（预估）：**
- 10个并发用户：请求量减少约 60%（缓存）
- 100个并发用户：后端压力显著降低
- 读多写少场景：GET 限流放宽，用户体验提升

---

### 5. 后续优化建议

1. **批量查询接口**
   - 合并多个小查询为一个大查询
   - 减少请求数量

2. **WebSocket 实时推送**
   - 关键数据变更时主动推送
   - 避免轮询

3. **后端数据库优化**
   - 添加索引加速查询
   - 连接池配置优化

4. **CDN 缓存**
   - 静态资源走CDN
   - 减轻服务器带宽压力

5. **Redis 分布式缓存**
   - 多实例共享缓存
   - 支持更高并发
